Project 1 — CLI AI Agent (Reflection)
Overview

In this project, I built a command-line AI agent capable of interacting with external tools (web search and file reading) using a tool-calling loop. The agent supports multi-turn conversations, executes tools dynamically, and feeds results back into the model for reasoning.

The goal was to understand the core architecture of AI agents:
observe → decide → act → observe → repeat

What I Built
A CLI-based AI assistant using a Groq-hosted LLM (OpenAI-compatible API)
Tool calling system with two tools:
Web search (via DuckDuckGo)
Local file reader
Full agent loop that:
Sends messages to the model
Detects tool calls
Executes tools
Feeds results back into the model
Multi-turn conversation memory using message history
Error handling for tool execution failures
Core Concepts Learned
How LLM tool calling works in practice
Designing an agent loop from scratch without frameworks
Managing conversation state across multiple turns
Structuring tools using JSON schema
Integrating external systems (search + filesystem) into LLM reasoning
Debugging agent behavior when tool outputs affect model decisions
Key Architecture Insight

The most important takeaway was that an “agent” is not a single model call, but a loop:

Send conversation + tools to the model
Model decides whether to respond or call a tool
If tool is called:
execute tool in Python
return result back into conversation
Repeat until final answer is produced

This loop is the foundation of all modern agent frameworks.

Challenges I Faced
Handling tool call parsing correctly
Ensuring tool outputs were properly formatted for the model
Managing conversation memory across multiple turns
Understanding differences between OpenAI-compatible tool calling and Anthropic-style tool use
What I Would Improve Next
Add persistent memory (saving chat history to disk)
Improve tool output formatting for better reasoning
Add more tools (calculator, database query, etc.)
Experiment with Anthropic’s native tool-use format for comparison
Final Reflection

This project helped me understand that agent systems are fundamentally about orchestration, not model complexity. The real engineering work is in managing state, tools, and control flow around the model—not just calling an API.
